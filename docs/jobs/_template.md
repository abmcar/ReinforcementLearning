## 背景

用 2-4 句话说明这个 job 在整体项目里的位置、要解决什么问题、为什么需要它。不要重复 `docs/roadmap.md` 已经说过的内容,只补充与本 job 相关的上下文。

## 目标

用 1-2 句话说明本 job 的**核心交付**。能用"完成 ××,使得 ××"句式表达最好。

## 输入

- **依赖产出**:列出依赖 job 已经产出的文件/模块路径,例如 `src/data/split.py` 提供 `load_split()`。
- **数据**:本 job 需要读取的数据文件路径和字段。
- **配置约定**:已经存在的配置文件、命名约定、目录结构。

## 工作内容

列出 3-7 个**结果导向**的工作项(不是详细步骤)。例如:

- 实现 X 模块,接口为 `def foo(...) -> Bar`
- 跑通 Y 流程,产出 `outputs/zzz/`
- 写一份 `docs/xxx.md` 说明 ××

避免:
- 把每一行代码怎么写规定死(agent 应该有判断空间)
- 重复已经在 roadmap 里说过的技术选型(只在本 job 特别需要时再强调)

## 产出

明确列出本 job 完成后**必须提交到 Git 的文件**(代码、配置、文档、图表),以及**不应该提交**的产物(模型权重、原始日志,放 `outputs/` 或 `runs/`)。

## 验收标准

可勾选的检查项,每条都应该是**客观可验证**的:

- [ ] 代码能在本地从干净环境跑通(给出命令)
- [ ] 单元测试 / 冒烟脚本通过
- [ ] 关键指标 / 结果落到 `docs/` 里

## 参考资料

- 论文 / 博客 / 代码仓库的 URL,**只列与本 job 强相关的 1-3 条**。
- 项目内文档(`docs/xxx.md`)的相对链接。
