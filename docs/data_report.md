# JOB-01 数据探索 (EDA) 统计报告

## 1. 数据规模与时间范围
* **Worker 数据**：包含约 1800 个众包工人的质量分数评估。
* **Project 数据**：包含约 2500 个项目的基本信息（分类、行业、起始与截止时间）。
* **Entry 数据**：包含超 22,800 条具体的工人提交记录。

## 2. 关键图表与分布发现
以下图表均保存在 `docs/figures/` 目录下：
1. **![Worker质量分布](./figures/JOB-01-worker_quality_distribution.png)**
   * **发现**：工人质量分数在经过非负过滤和归一化后，呈现典型的长尾/偏态分布，部分优秀工人的分数集中在高分段。
2. **![项目提交数分布](./figures/JOB-01-project_entry_count_distribution.png)**
   * **发现**：绝大多数项目收到的作品提交数 (Entry Count) 集中在 50 次以内，数据存在稀疏性。
3. **![项目周期分布](./figures/JOB-01-project_duration_boxplot.png)**
   * **发现**：大部分项目的生命周期在几天到数周不等，存在少量极端长周期的异常项目。

## 3. 对下游任务 (JOB-02 / JOB-04 / JOB-07) 的建议
1. **冷启动与稀疏性 (给 JOB-02/06)**：由于大部分项目收到的提交较少，推荐系统在召回和划分数据集时，必须考虑长尾 Project 的冷启动问题。
2. **字段异常 (给 JOB-03)**：在处理 entry 数据时，JSON 实际使用的键名是 `worker` 而不是大纲里声称的 `author`，特征工程提取时需格外注意。
3. **奖励函数 (给 JOB-07)**：worker 的质量分数包含 <= 0 的异常值，在构建 RL 的 Requester Reward 时，必须沿用本脚本的过滤和归一化逻辑，避免输入负数导致 Q 值发散。
